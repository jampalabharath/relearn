Copy Activity

Copy Activity in Azure Data Factory

What is Copy Activity?

Copy Activity is the core data movement activity in Azure Data Factory (ADF). It allows you to securely and efficiently copy data from a source data store to a destination (sink) data store. This is widely used for data ingestion, ETL pipelines, and data migration across on-premises, cloud, and hybrid environments.

Key Features

  • Supports 100+ connectors for structured, semi-structured, and unstructured data.

  • Scalable data movement ‚Äì can handle gigabytes to terabytes of data.

  • Secure integration with managed identities, Key Vault, and encrypted channels.

  • Rich transformation support like column mapping, schema drift handling, and format conversions.

  • Parallelism & partitioning for high-performance copying.

Components of Copy Activity

  1. Source (From) ‚Äì The dataset or linked service where the data resides (e.g., SQL DB, Blob storage, REST API).

  2. Sink (To) ‚Äì The destination dataset where the data will be written (e.g., ADLS, Synapse, Cosmos DB).

  3. Mapping ‚Äì Defines how columns/fields from the source map to the sink.

  4. Settings ‚Äì Performance tuning options such as degree of parallelism, staging, and fault tolerance.

Workflow of Copy Activity

  1. A pipeline triggers the Copy Activity.

  2. ADF reads data from the source using a linked service.

  3. Data is optionally transformed (format, schema mapping, partitioning).

  4. Data is written to the sink.

  5. The process is logged and monitored in the ADF monitoring dashboard.

Supported Data Stores

  • Azure Sources/Sinks: Blob Storage, Data Lake (ADLS Gen1/Gen2), Azure SQL, Synapse, Cosmos DB, Table Storage.

  • On-Premises: SQL Server, Oracle, SAP, File system (via Self-hosted IR).

  • SaaS Applications: Salesforce, Dynamics 365, ServiceNow, Google Analytics, etc.

  • Big Data & NoSQL: Amazon S3, MongoDB, Cassandra, Hadoop.

Performance Tuning Options

  • Parallel Copy: Enable parallelism to copy partitions of data simultaneously.

  • Staging: Use Azure Blob/ADLS as a staging area for complex transformations.

  • Batch Size & Block Size: Adjust for optimal throughput.

  • Self-hosted IR Scaling: Scale out compute for large hybrid data transfers.

Monitoring & Troubleshooting

  • Activity Runs View: Shows status (Success, Failed, In Progress).

  • Detailed Metrics: Throughput, rows read/written, time taken.

  • Retry Policies: Configure automatic retries on transient failures.

  • Integration with Log Analytics: Centralized monitoring and alerting.

Example Use Cases

  • Migrating on-prem SQL Server data to Azure SQL Database.

  • Ingesting CSV/JSON/XML files from Blob Storage to Synapse for reporting.

  • Copying SaaS application data (e.g., Salesforce) into Data Lake for analytics.

  • Hybrid ETL: Moving data from on-prem Oracle DB to ADLS for big data processing.

Best Practices

  • Use Integration Runtime close to data sources for better performance.

  • Leverage incremental loads instead of full loads to optimize costs.

  • Enable fault tolerance for large jobs.

  • Store sensitive connection strings in Azure Key Vault.

  • Use parameterized datasets for reusability.

üìÇ Source Options for ADLS in Copy Activity

1. File Path Settings

  • File path type ‚Üí Choose how to select files:

    • File path ‚Üí Specific file (container/folder/file).

    • Wildcard file path ‚Üí Use * or ? to match multiple files. Example: input/*.csv.

    • Folder path ‚Üí Copy all files in a folder.

  • Recursive ‚Üí If enabled, copies files from subfolders as well.

2. File Filter Options

  • File name wildcard ‚Üí Filter by pattern (e.g., *.json, 2025-09-*.parquet).

  • Modified date filter ‚Üí Pick files based on last modified time. Useful for incremental loads.

  • List of files ‚Üí Provide explicit list of files (parameterized).

3. Binary Copy

  • Binary Copy Mode ‚Üí If enabled, files are copied as-is (no parsing). Used for images, videos, executables.

4. Compression

  • If files are compressed, choose:

    • Gzip, Bzip2, Deflate, Zip ‚Üí Auto-decompress on read.

    • None ‚Üí If files are plain text or structured formats.

5. File Format Settings (if not binary copy)

  • Delimited Text (CSV/TSV):

    • Row delimiter, column delimiter

    • Escape character, quote character

    • First row as header (yes/no)

  • JSON:

    • Single object or Array of objects

  • Parquet / ORC / Avro:

    • Schema auto-detection

    • Support for column projection

6. Partition Options (for performance)

  • Max concurrent connections ‚Üí Number of parallel threads to read files.

  • Partition by folder/file ‚Üí ADF can split work by file or by folder for faster ingestion.

7. Additional Options

  • Preserve hierarchy ‚Üí Decide how to keep folder structure when moving data:

    • PreserveHierarchy ‚Üí Keeps folder structure.

    • FlattenHierarchy ‚Üí Drops folder paths, copies only files.

    • MergeFiles ‚Üí Combines multiple files into a single output file.

  • Ignore empty files ‚Üí Skip empty files instead of failing.

  • Max concurrent connections ‚Üí Speed up large dataset ingestion.

‚úÖ Example Scenarios:

  • Incremental ingestion: Use last modified filter and wildcards (2025-09-16*.csv).

  • Big Data migration: Enable recursive + parallel reads + PreserveHierarchy.

  • Data Lake to Synapse pipeline: Use Parquet/CSV reader with schema mapping.

üóÑÔ∏è Source Options for SQL Server in Copy Activity

1. Data Retrieval Options

  • Table ‚Üí Pick a specific table from the connected database.

  • Query ‚Üí Provide a custom SQL query (parameterized if needed).

  • Stored Procedure (only in some cases) ‚Üí Call a stored procedure and use its result set as source.

2. Query Behavior

  • Use Query from Dataset or Inline ‚Üí You can store SQL queries in the dataset definition or write inline queries in the activity.

  • Parameterization ‚Üí Pass pipeline parameters into the query (e.g., SELECT * FROM Sales WHERE OrderDate = '@{pipeline().parameters.Date}').

3. Batching & Fetch Options

  • Batch Size ‚Üí Number of rows fetched in one go (default varies, typically ~10,000). Helps tune performance for large datasets.

  • Isolation Level ‚Üí Read Uncommitted, Read Committed, Snapshot, etc. (controls concurrency & locking).

4. Partition Options (for parallel reads)

  • Partition Option ‚Üí Split source data into multiple partitions for faster reads. Options:

    • None ‚Üí Single-threaded copy.

    • Physical Partitions ‚Üí Use table partitions (if available).

    • Dynamic Range ‚Üí Partition data based on numeric/date column (e.g., OrderID BETWEEN 1‚Äì10000, etc.).

  • Partition Column ‚Üí Column used for partitioning (usually numeric or date).

  • Number of Partitions ‚Üí Controls parallelism (e.g., 4 partitions = 4 parallel queries).

5. Additional Options

  • SQL Command Timeout ‚Üí Maximum time allowed for query execution.

  • Detect Changes (Incremental Load) ‚Üí Use watermark columns (like LastModifiedDate) for incremental copies.

  • Enable CDC (Change Data Capture) ‚Üí If SQL Server has CDC enabled, ADF can use it to capture only changed rows.

6. Data Consistency & Validation

  • Table/Query Validation ‚Üí Validate schema before copying.

  • Null Handling ‚Üí How NULL values are handled when mapping.

  • String Handling ‚Üí Options for trimming, padding, or encoding.

‚úÖ Example Use Cases

  • Full table load ‚Üí Choose Table option.

  • Incremental load ‚Üí Use Query with a WHERE LastModified > @{pipeline().parameters.LastRunTime} filter.

  • High-volume migration ‚Üí Use Dynamic Range partitioning on an indexed column for parallel ingestion

1Ô∏è‚É£ Sink: CSV (Delimited Text in Blob/ADLS)

When you choose CSV as the sink format, you‚Äôll see file-related settings.

File Path Options

  • File name option

    • Output to single file ‚Üí All data is written into one file.

    • Per partition/file ‚Üí Each partition creates a separate file (e.g., part-0001.csv).

    • Preserve source hierarchy ‚Üí Keeps folder/file structure from source.

  • File naming pattern ‚Üí Custom names like data_{date}.csv.

  • Folder path ‚Üí Target container/folder in Blob/ADLS.

File Format Settings

  • Row delimiter ‚Üí \n (default), \r\n, or custom.

  • Column delimiter ‚Üí , (default), ;, |, \t, etc.

  • Quote character ‚Üí Usually ".

  • Escape character ‚Üí Usually \.

  • Null value representation ‚Üí For example, empty string, \N, or "NULL".

  • Write headers ‚Üí Yes/No (include column names as first row).

Compression

  • None, Gzip, Bzip2, Deflate, Zip.

Copy Behavior

  • Overwrite (replace existing files).

  • Append (add new file without deleting old ones).

  • Merge (combine into existing file ‚Äì limited scenarios).

2Ô∏è‚É£ Sink: SQL Server

When SQL Server is the sink, the options are more database-related.

Table & Write Options

  • Table name ‚Üí Select or provide the target table.

  • Auto-create table ‚Üí ADF can auto-generate the table if it doesn‚Äôt exist (based on source schema).

  • Pre-copy script ‚Üí SQL script that runs before insert (e.g., TRUNCATE TABLE TargetTable).

  • Stored procedure sink ‚Üí Instead of direct inserts, call a stored procedure with input parameters.

Write Behavior

  • Insert ‚Üí Default (append new rows).

  • Upsert ‚Üí Insert new + update existing (requires key column).

  • Update ‚Üí Update only existing rows.

  • Stored procedure ‚Üí Custom merge logic via proc.

Batching & Performance

  • Batch size ‚Üí Number of rows per batch insert.

  • Bulk insert (PolyBase/BCP) ‚Üí Faster loading for large volumes.

  • Table lock ‚Üí Lock the table during write (improves speed but reduces concurrency).

Additional Options

  • Identity insert ‚Üí Control whether identity columns are written or auto-generated.

  • Null handling ‚Üí Define how nulls map to SQL columns.

  • Transaction settings ‚Üí Commit mode (per batch or per entire load).

  • SQL command timeout ‚Üí Max allowed time for inserts.

‚úÖ When to use what?

  • CSV Sink ‚Üí Good for data lake staging, archival, or feeding BI/ML pipelines.

  • SQL Server Sink ‚Üí Good for operational databases, reporting, or direct analytics.
